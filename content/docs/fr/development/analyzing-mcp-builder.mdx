# Analyse de mcp-builder : Un Guide Complet pour le Développement de Serveurs MCP

**mcp-builder** est un skill Claude qui fournit un guide complet pour créer des serveurs MCP (Model Context Protocol) de haute qualité. Ce skill permet aux LLM d'interagir avec des services externes via des outils bien conçus, prenant en charge les implémentations Python (FastMCP) et Node/TypeScript (MCP SDK).

<Callout type="info">
  Il s'agit d'un skill prêt pour la production provenant du dépôt de skills Anthropic, conçu pour guider les développeurs tout au long du cycle de vie complet du développement de serveurs MCP, de la planification à l'évaluation.
</Callout>

## Vue d'ensemble

### Qu'est-ce que mcp-builder ?

<Callout type="question">
  Basé sur la description : Guide pour créer des serveurs MCP (Model Context Protocol) de haute qualité qui permettent aux LLM d'interagir avec des services externes via des outils bien conçus. À utiliser lors de la construction de serveurs MCP pour intégrer des API ou services externes, que ce soit en Python (FastMCP) ou Node/TypeScript (MCP SDK).
</Callout>

### Objectif principal

Le skill mcp-builder vise à :

- Guider les développeurs dans la conception de serveurs MCP centrés sur l'agent
- Fournir des patterns d'implémentation complets pour Python et TypeScript
- Établir des pratiques de développement pilotées par l'évaluation
- Créer des outils optimisés pour les limites de contexte des LLM
- Permettre l'intégration systématique de services externes

### Public cible

Ce skill est conçu pour :

- Les développeurs construisant des serveurs MCP pour l'intégration d'API externes
- Les équipes créant des outils réutilisables pour Claude et d'autres LLM
- Les ingénieurs implémentant les spécifications du Model Context Protocol
- Toute personne intéressée par les patterns de communication LLM-service externe

### Phase 4 : Création d'évaluations

L'aspect le plus sophistiqué de mcp-builder est son framework d'évaluation :

#### 4.2 Créer 10 questions d'évaluation

Chaque question doit répondre à six exigences strictes :

1. **Indépendante** : Ne dépend pas d'autres questions
2. **Lecture seule** : Nécessite uniquement des opérations non destructrices
3. **Complexe** : Nécessitant plusieurs appels d'outils et une exploration approfondie
4. **Réaliste** : Basée sur des cas d'usage réels qui intéressent les humains
5. **Vérifiable** : Une réponse unique et claire qui peut être vérifiée par comparaison de chaînes
6. **Stable** : La réponse ne changera pas avec le temps

#### 4.4 Format de sortie

Les évaluations créent des fichiers XML avec cette structure :

```xml
<evaluation>
  <qa_pair>
    <question>Find discussions about AI model launches with animal codenames...</question>
    <answer>3</answer>
  </qa_pair>
  <!-- Plus de qa_pairs... -->
</evaluation>
```

## Analyse des scripts

### connections.py - Gestionnaire de connexions MCP multi-transport

Ce module Python de 152 lignes fournit une abstraction de connexion sophistiquée :

```python
class MCPConnection(ABC):
    """Classe de base pour les connexions de serveurs MCP."""

    async def __aenter__(self):
        """Initialiser la connexion au serveur MCP."""
        # Gère AsyncExitStack, l'initialisation de session et le nettoyage des erreurs
```

**Caractéristiques clés** :
- Pattern de classe de base abstraite pour une interface cohérente
- Prise en charge de trois protocoles de transport : stdio, SSE, HTTP
- Gestionnaire de contexte asynchrone pour le nettoyage des ressources
- Initialisation automatique de session et gestion d'erreurs
- Fonction de fabrique `create_connection()` pour une instanciation indépendante du transport

**Classes de transport** :
- `MCPConnectionStdio` : Entrée/sortie standard pour les serveurs locaux
- `MCPConnectionSSE` : Server-Sent Events pour les connexions en streaming
- `MCPConnectionHTTP` : HTTP diffusible pour les serveurs MCP basés sur le web

### evaluation.py - Cadre d'évaluation complet pour serveurs MCP

Ce module de 374 lignes fournit des capacités d'évaluation de bout en bout :

<Callout type="info">
  Le framework d'évaluation teste si les LLM peuvent utiliser efficacement les serveurs MCP pour répondre à des questions réalistes et complexes.
</Callout>

**Composants principaux** :

1. **Invite d'évaluation** : Invite système sophistiquée nécessitant :
   - Utilisation d'outils avec des résumés étape par étape
   - Retours constructifs sur la conception des outils
   - Réponses XML correctement formatées

2. **Analyse XML** : Analyse robuste des fichiers d'évaluation avec gestion d'erreurs

3. **Boucle agent** : Interaction asynchrone avec l'API Claude et les outils du serveur MCP

4. **Collecte de métriques** : Suivi complet de :
   - Taux de précision
   - Durées des tâches
   - Nombre et performance des appels d'outils
   - Qualité des retours

**Exemple d'utilisation** :

```bash
# Évaluer un serveur MCP stdio local
python evaluation.py -t stdio -c python -a my_server.py eval.xml

# Évaluer un serveur MCP SSE avec authentification
python evaluation.py -t sse -u https://example.com/mcp \
  -H "Authorization: Bearer token" eval.xml
```

## Pièges courants

<Callout type="warn">
  Erreurs courantes commises par les développeurs lors de la construction de serveurs MCP
</Callout>

### Conception API-First (approche incorrecte)

**Symptôme** : Les outils mappent 1:1 avec les points de terminaison d'API

**Problème** : Force les agents à comprendre la structure de l'API et à faire plusieurs appels pour des workflows simples

**Solution** : Concevoir des outils autour des workflows et des tâches, pas des points de terminaison d'API

### Retour d'informations excessif

**Symptôme** : Les outils retournent des réponses API complètes avec tous les champs

**Problème** : Gaspille la fenêtre de contexte avec des données non pertinentes

**Solution** : Ne retourner que les informations à fort signal, fournir des options détaillées/concises

### Messages d'erreur médiocres

**Symptôme** : "Error 404: Not Found"

**Problème** : Les agents ne savent pas comment récupérer

**Solution** : "Resource not found. Essayez d'utiliser list_resources() pour voir les options disponibles"

### Annotations d'outils manquantes

**Symptôme** : Les outils manquent `readOnlyHint`, `destructiveHint`, etc.

**Problème** : Claude ne peut pas optimiser les patterns d'utilisation des outils

**Solution** : Ajouter des annotations appropriées à tous les outils

### Tests insuffisants

**Symptôme** : Pas de framework d'évaluation, tests manuels uniquement

**Problème** : Impossible de mesurer l'efficacité des outils ou d'itérer basé sur les retours

**Solution** : Créer des évaluations complètes en utilisant evaluation.py de mcp-builder

## Dépannage

### Serveur MCP bloqué au démarrage

**Symptôme** : Le serveur démarre mais le processus reste bloqué indéfiniment

**Cause** : Les serveurs MCP sont des processus longue durée qui attendent des requêtes

**Solution** : Exécuter dans tmux ou utiliser un timeout pour les tests :
```bash
timeout 5s python server.py
```

### Échecs de connexion du framework d'évaluation

**Symptôme** : evaluation.py ne peut pas se connecter au serveur MCP

**Cause** : Inadéquation du protocole de transport ou problèmes d'authentification

**Solution** :
- Vérifier le type de transport correct (stdio/SSE/HTTP)
- Vérifier les informations d'identification d'authentification pour les serveurs distants
- S'assurer que le serveur est en cours d'exécution avant de lancer l'évaluation

### Échecs d'appels d'outils dans l'évaluation

**Symptôme** : Les outils s'exécutent mais retournent des erreurs

**Cause** : Échecs de validation des entrées ou modifications d'API

**Solution** :
- Réviser les schémas d'entrée des outils pour une validation appropriée
- Vérifier que les points de terminaison d'API sont accessibles et inchangés
- Vérifier que la gestion d'erreurs fournit des retours actionnables

### Scores d'évaluation médiocres

**Symptôme** : Faible précision sur les questions d'évaluation

**Cause** : Les outils ne sont pas conçus pour les workflows d'agents ou la documentation est médiocre

**Solution** :
- Réviser les principes de conception centrés sur l'agent
- Améliorer les descriptions des outils avec de meilleurs exemples
- Consolider les opérations liées en outils de workflow
- Ajouter une gestion d'erreurs complète et des conseils

### Problèmes de dépassement de contexte

**Symptôme** : Les outils retournent trop de données, dépassant les limites de contexte

**Cause** : Pas de stratégie de pagination ou de troncature

**Solution** :
- Implémenter des paramètres de pagination
- Ajouter des vérifications de limites de caractères
- Fournir des options de réponse concises/détaillées
- Utiliser le plafond de `25 000` jetons comme guide

### Fondamentaux du protocole MCP

MCP (Model Context Protocol) permet aux LLM d'interagir avec des services externes via :

- **Outils** : Fonctions que les LLM peuvent appeler avec des entrées structurées
- **Ressources** : Données qui peuvent être chargées dans le contexte
- **Invites** : Modèles d'invites réutilisables
- **Transport** : Mécanisme de communication (stdio, SSE, HTTP)

### Métriques de qualité de l'évaluation

Le framework d'évaluation suit :

- **Précision** : Réponses correctes / questions totales
- **Performance** : Durée moyenne des tâches et efficacité des appels d'outils
- **Utilisation des outils** : Nombre et durée des appels d'outils par tâche
- **Qualité des retours** : Problèmes signalés par l'agent et suggestions d'amélioration

Jalons typiques :
- **Bon** : &gt;80 % de précision, &lt;30 s de durée moyenne
- **Excellent** : &gt;90 % de précision, &lt;20 s de durée moyenne
- **Exceptionnel** : &gt;95 % de précision, &lt;15 s de durée moyenne
